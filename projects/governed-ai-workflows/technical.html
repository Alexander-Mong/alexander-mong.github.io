How it works

The model does two small jobs. The engineering is everything that makes them trustworthy.

Ask a language model for "the key insights" from a stack of transcripts and you get a confident list — with no way to tell what it invented, what it dropped, or whether any of it is even in the source. This pipeline answers a narrower, checkable question: which findings are real, and can each be traced back to the exact words it came from? The model does a small, bounded job; the work is the apparatus around it that re-checks every call against the source instead of trusting it. That discipline runs deep enough that a junk control I pre-registered to fail beat both of my production lenses — the first of the designs I falsified, kept as a record of the system catching my own mistake.

The design fights two failure modes. The first is hallucination: the model asserting something the source never says. The second is impressionistic selection: the model "choosing" what matters by feel, in a way you can't measure or repeat. So the work is split. Code handles everything you can count or anchor to a byte. The model does the two things code can't: it finds what the fixed rules missed, and it judges whether a candidate is real. A person can overrule, and grades samples to sharpen the judge.

The code does the mechanical work: it ingests and segments the sources, assigns content-addressed IDs, anchors every span to byte offsets and a timecode, matches versioned rules and routes candidates, counts and dedupes, gates on a fixed schema, and re-resolves every promoted quote to its hashed source. The model does two things, both bounded: it judges whether a located candidate is really an X (yes, no, or uncertain), and in the coverage check it surfaces real items the fixed rules missed. Both run against a written definition that's still under test, and every call is re-anchored and re-checked in code. The model never anchors a span, counts anything, or invents text the source doesn't contain. The code establishes all of that.

Where the model decides — and where it doesn't

One diagram for the whole system. Source forks into two finders — deterministic rules and an LLM lens — that fold into a single pool; the model casts one bounded vote on each pooled candidate; keepers anchor to source and land in a traceable result. The cobalt marks are the only places a model touches the work — note how little of the surface they own. (This isn't retrieval or RAG: the model never ranks or retrieves over the corpus. As a finder it reads one whole talk at a time; as the judge it casts a single bounded vote on a pooled candidate.)

Every promoted finding resolves byte-for-byte to its hashed source

A private corpus can't offer "run it yourself." Traceability is the substitute: a promoted finding isn't a sentence the model produced, it's a pointer — through a chain of named, version-pinned steps — to a byte-exact span inside a hash-locked transcript. A blocking validator recomputes the source hash and asserts the quoted text still resolves byte-for-byte at its recorded offsets; it runs over a run's artifacts and in the test suite, so a drifted byte fails loudly instead of shipping silently. Anti-fabrication here isn't a promise; it's an assertion in code. (Running that same check inside the promotion projection on every pass, rather than as a separate gate, is a named next step — see the gaps below.)

The check is small and dull on purpose: a validator that recomputes the source digest, re-slices the recorded offsets, and asserts the bytes are identical. Because it is dull, it runs the same way every time. In its essential shape, the check is these few lines:

Validator — the invariant it enforces (sanitized excerpt) # A promoted finding must still resolve byte-exact to its hashed source. # Run as a blocking validation check; a drift fails loudly, it does not warn. def assert_resolves(finding, source) -> None: digest = sha256(source.bytes) assert digest == finding.source_hash, "source changed under a promoted finding" start, end = finding.span # byte offsets recorded at Stage 1 quoted = source.bytes[start:end].decode("utf-8") assert quoted == finding.quote, "quote no longer matches the anchored span"

Excerpt rewritten to its essential shape for display; identifiers and the real call sites stay private. The contract — recompute the hash, re-slice the offsets, assert equality — is the actual one. This is deliberately the cheap insurance policy: a few lines that can never be talked out of failing, bolted on top of the expensive deterministic layer below — versioned rule-matching, routing, dedup, and the schema gate that decides what is even eligible to reach it.

The validator is the small end. The deterministic surface around it has real structure: rules are named and versioned so a finding records exactly which definition produced it, and a finding is checked against a fixed schema — an anchored span, a named pinned rule, a recorded vote — before it is eligible at all. Eligibility is the schema check; selection is a separate, deliberately simple rule (sanitized):

Eligibility schema + selection (sanitized shape) # Each rule carries an id + version; bumping the rule bumps the version, # so a stored finding always names the exact definition that fired. Rule(id="rule_<name>", version="v2", match=…, context_chars=120) # Eligible = structurally complete: an anchored span, a named pinned # rule, and a recorded judge vote. Missing any one drops the finding. def is_eligible(f) -> bool: return ( f.span is not None # anchored at Stage 1 and f.rule_version in KNOWN_RULES # a named, pinned rule and f.vote is not None # a judge actually voted ) # Selection is single-lane: keep what the precision-lane judge confirmed. # A second vendor's agreement is recorded as a confidence label, not a gate. def selected(f) -> bool: return is_eligible(f) and f.precision_vote == "confirm"

The hard deterministic work (versioned rule-matching, routing, dedup, cost-gating, and the schema check) is the real surface area here, not the LLM call. Selection is intentionally single-lane — one judge's confirm — with cross-model agreement and the byte-exact validator layered on as a confidence signal and a blocking check, not as the thing that auto-promotes. What reaches the dashboard is a triage-grade set, ranked by confirm-vote count. The excerpts are small on purpose: what matters is the invariant they enforce, not the line count.

The cross-vendor signal is a confidence label, not a gate: when the two probe vendors split, the item is still selected by the single production vote — it's just flagged lower-confidence (“mixed”) and a prime candidate for the human's attention. Agreement never certifies truth on its own anyway: two vendors can share a blind spot (see the κ field notes).

Three designs I falsified — including a junk decoy that outscored production

These are three things I tried, measured, and killed when the data went against them. I keep them as dated records, next to the designs they replaced. They are really one story: my attempts to score each finding's "value" on a single scale. The naive scorer was falsified, its replacement retracted, and the ranking loop withdrawn — one idea, killed in stages.

1. A junk decoy outscored both production lenses (falsified). To test a value-scoring step, I pre-registered a deliberately worthless "lens" — one that grabs a shallow surface feature and never real content — as a floor any real scorer should beat. On the first design it didn't lose: the decoy ranked above both of my production lenses, because the scorer was rewarding surface features instead of substance. I killed that naive judge rather than patch it, and left the decoy in as a permanent floor. The direction held on a rebuild; only the exact margin was a small-sample artifact, retracted below. No clean replacement value scorer ever passed, so what actually floors the decoy now is a different measurement: the advice extractor's verbatim-confirmation rate, about 71% (n=268) against the control's 2.5% (n=120), blind-graded on a 17-transcript sample.

2. A "clean" separation that vanished when I scaled the sample (retracted). A value-panel comparison passed at a small per-cell sample, and the decoy looked safely floored. Re-run with more data, the separation dissolved: the clean version was a small-sample artifact, not a finding. I retracted it, published the retraction next to the original, and made it a standing rule — repeat and scale before believing any separation, especially a flattering one.

3. The per-lens "value" score didn't hold, so I withdrew the loop (retracted). An early loop scored each item's usefulness on one scale and ranked the ambiguous middle. The clearly-junk and clearly-strong items separated reliably across blind passes, but the muddy middle never ranked stably. So I cut the loop rather than ship a number I couldn't defend, and kept only what the evidence supported: flag the clearly-low and clearly-high, and don't rank the middle.

What these forced is the rule the rest of the system runs on: measure the objective signals you can prove, estimate the subjective ones you can't, and don't collapse the two into a single score.

What the cheap reliability checks miss

Running the judging as a real measurement turned up a few things that aren't obvious going in. Each one is tagged with how far I'd trust it.

Asking the same model twice doesn't catch its confident mistakes (directional, small sample)

Asking the same model the same thing K times and trusting the agreement feels rigorous, but it can't catch the errors that matter. A confidently-wrong model agrees with itself every time, so the mistakes you most need to find are the ones this method is blind to. No K fixes that; it's structural. Over a 30-span ladder it caught 0 of its 2 known confident-error cases, because the model was unanimously wrong on both. To catch those you need an independent model that can disagree, not a repeat of the same one.

Two vendors are not two independent votes (validated — Cohen's kappa)

The fix for confident errors is a second, independent judge. But two models can agree because they share a blind spot, so agreement alone isn't proof. GPT and Gemini agree at Cohen's κ = 0.74 on the 39-item hand-labeled set, and on a broader 108-item unlabeled pool that drops to κ = 0.53. At the low end that is barely more than one independent vote. A second model helps, but the reliability comes from the measurement, not from adding another logo.

How those numbers are derived: the 0.74 carries a bootstrap CI of roughly [0.50, 0.95]; the 0.53 is a point estimate with no CI. Across all three model judges (GPT, Gemini, Claude), Fleiss' κ is 0.69 on the labeled set and 0.61 on the pool — that Fleiss is over the three models, not the human hand-labels, which are the separate gold set the accuracy is scored against. The samples are small, so read every κ as an estimate. Recomputed from stored verdicts on 2026-06-20.

More context isn't better — accuracy peaks in a narrow window (directional, small sample)

Feeding the judge more surrounding text made it less accurate, not more. Accuracy peaked around ~120 characters of context and fell off toward 800; the extra context mostly adds distractors. The effect is per-task — some construct types are hurt by more context, others have a sweet spot — so the lesson is to measure each task type instead of picking one global setting.

The tags aren't decoration: the κ is reproduced Cohen's kappa, so it's marked validated; the catch-rates and the context curve are directional on small samples, so they say so.

The loop refused to promote its own change

The system improves itself, but only through a human gate and an honesty check. Verdicts and coverage feed back into the rules and the judge's definitions; nothing is applied automatically — a person reviews each proposed change before it lands. A separate distillation experiment tried to sharpen the judge and promote the result only if it was strictly better on a held-out set the change had never seen.

On its first run, that gate refused its own candidate: the new judge wasn't strictly better on the held-out data, so it never shipped. A loop that can't decline a worse candidate would just drift, so the refusal matters — and it's logged on the same immutable record as the falsifications above, rather than quietly overridden.

The calls I made, and what each one gave up

Each of these was a real tradeoff. Here is what I chose, and what it cost:

The callChoseInstead ofWhat I gave up
How to structure the judgment Explicit, inspectable gates between stages One end-to-end prompt that "does it all" Convenience and fewer moving parts — for a pipeline where every decision is locatable
What gets into the result set Promote only on a controlled judge-confirm; a human can overrule Auto-accept the model's raw output Throughput — for the ability to stand behind every promoted item
What the source corpus is A fixed, hash-locked private corpus A live or growing source Freshness and "run it yourself" — for reproducibility and a stable provenance chain
How to spend on model calls Batch bounded judgments per construct, no agent harness One rich call per item Some per-item richness — for far cheaper runs with no accuracy cliff

On cost: generation runs on a flat monthly subscription — a real fixed cost, not free — with no metered per-call API spend. The only metered charges were two small cross-vendor probes (GPT/Gemini); the binding constraint throughout was sharper definitions, not more expensive machinery.

What this isn't, and where it's weak

This is not production NLP, a RAG system, a trained or deployed model, or "AI engineering." It is the evaluation and governance work around a model. The corpus stays private under NDA while the method is public — the data is the client's; the technique is mine to show.

Corpus counts are a dated snapshot, not eternal facts · as of June 2026

Any corpus count is frozen at the moment it's taken and moves as the corpus grows, so where a number appears in this project's materials it carries a date and is read as a snapshot, not a permanent property of the system.

The honest gaps, and where each goes next:

No true Stage-5 promoter. The final promotion currently projects from the confirmed set; a dedicated promoter that consumes the promotion schema isn't built. The schema exists and is unconsumed, which is the next concrete step.

Mid-tier value is muddy. The system reliably flags clearly-low and clearly-high usefulness but can't rank the ambiguous middle — the value-scoring arc I retracted above. For now it ships the honest two-bucket flag and leaves the middle to human review.

A fragile cross-project seam. The cross-vendor adapters live outside the project and are reached by brittle relative-path math — a real dependency seam, documented rather than hidden. It's isolated to the two metered probes, not the core pipeline.

Small samples throughout the reliability work. The confident-error catch rates rest on tiny denominators (the self-consistency result is n=2 confident-error cases over a 30-span ladder), and the one bootstrap CI shown is on the n=39 labeled set and is wide, [~0.50, 0.95]; the n=108 figure is reported without a CI. The direction and the mechanism are sound; the magnitudes need more cases. Every such claim is tagged directional / small-n on this page.

What holds today is the spine: a model confined to a narrow, bounded vote; every promoted finding pinned to a byte-exact span in a hashed source; and a record of the designs I tried and killed. The corpus and the full source stay private, but the method, the invariants, and the calls I made are laid out here.
