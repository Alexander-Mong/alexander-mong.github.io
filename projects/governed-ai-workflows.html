Governed AI Workflows

An LLM will hand you a confident list of "insights." I built the workflow that tells you which ones are real — and which ones it made up.

I treat a model’s output as a claim to be proven, not trusted. Here, across 153 conference-panel transcripts (distilled from 161 harvested videos), that meant measuring how much of what it surfaced held up, catching what it fabricated, and tracing every survivor back to its source — with a human reviewing and able to overrule. This evaluation and governance work — not model-building — is what I’m strongest at.

The discipline

Measuring whether the output can be trusted

Calling an API and getting fluent text back is easy. The hard part is proving it: how much is real, how much is invented. So I built the extraction as an evaluation problem and ran it as one, scored against a rubric fixed before any result was seen.

Faithfulness — is each statement real? A cheap model scores every routed candidate against the rubric (LLM-as-judge), and a stronger model independently re-judges a pre-registered sample, giving a cross-model agreement check. A planted set of hard-negative control cases measures whether the judge can correctly reject junk — the test that it can say no, not just yes.

Coverage — what did it miss? A separate pass has an LLM read whole transcripts to surface genuine items the rule-based pass could never catch, every find verbatim-anchored to its source and graded blind against two controls, including an adversarial decoy built to catch a judge that says "yes" to everything.

Under the hood, deterministic code does everything that moves, anchors, counts, gates, and routes; the model does only bounded semantic judgment; and a local review dashboard is where the human reviews. The Claude judging ran on a flat monthly subscription — a real fixed cost, not free — with no metered per-call API spend (a couple of small cross-vendor probes aside); the binding constraint was sharper definitions, not more expensive machinery. The judge is never the last word: a human can override the model and record why.

The first thing this workflow produces has no model in it at all: a vocabulary map of the corpus — the terms distinctive to this domain versus ordinary English, with how often and how widely each is used. It is pure counting, so it can’t hallucinate; it only reports what is verbatim in the source. That deterministic floor — everything countable handled by code that can’t invent — is what justifies letting a model judge the rest.

Proof it works

The measurement held up — and so did the audit trail

The judge cleared a bar I fixed before seeing any result: 28 of 29 planted hard negatives correctly rejected. Keeper-tier findings were graded blind by four annotators across two vendors (three LLMs plus my own hand-grade), and 35 of 36 were also rated top-tier by another model. And it found what the rules alone couldn’t: a separate coverage pass turned up about 1,900 verbatim items the rules missed across the full corpus, with a blind sample estimating roughly 1,150 of them genuine (directional — a single-model sample, not yet a full human pass).

By construction, every promoted finding carries its full chain back to a hashed source span, with the method and version recorded at each hop. I walked one confirmed finding end-to-end as a spot-check, and the quoted text resolved byte-for-byte to its position in the hashed source.

Sources are publicly available conference-panel videos; transcripts were processed locally and are not reproduced here. Findings appear only as derived, aggregated counts.

Validating the judge before I relied on it

A judge is only useful if you can show it’s reliable — so before relying on it I calibrated it against a small set of hand-confirmed examples, including deliberately hard look-alikes built to make it fail, then tested it on fresh cases it had never seen. On the cases validated so far it held up on that unseen test; the calibration even caught one of my own judgment calls that was wrong; and where it still falls short, I say so. It also showed me what wasn’t worth spending on: the bottleneck was sharper definitions, not more expensive machinery.

The trace caught my own mistake

That same provenance walk caught a mistake in my own write-up before it went out — a claim about the data that the trace proved wrong. I corrected it. And a separate, promising early result didn’t survive a more rigorous re-run, so I retracted it and corrected the record. That is what I built the system to do: not to prove I don’t make mistakes, but to surface them — mine included.

Running it as a measurement turned up real findings

Because the judging was built and run as a real experiment, it surfaced results that aren’t obvious going in. Two examples: feeding the judge more surrounding context made it less accurate, not more, with a measurable sweet spot; and a model that agrees with itself across repeated runs is signalling confidence, not correctness — a confidently-wrong judge sails straight through a self-agreement check, so catching that class of error needs a second, independent model. And when I measured it, two vendors’ votes carried the weight of barely more than one independent vote, so a second model is necessary but not sufficient.

Scope and honesty

What I deliberately left out

Out of scope, by choice: a production deployment, a full human-review promotion pass, an automated A/B ablation, and a larger higher-confidence study. Each is a defined next step with a stated reason, not a loose end — I left them out to keep the method provable under the deadline I had.

What this demonstrates

The through-line

The through-line across everything I build: a messy real-world workflow turned into a structured, governed, traceable system whose output a human can stand behind.

Designing a rubric, judging model output against it, checking the judges against each other, planting adversarial cases, anchoring every claim to its source, and knowing when to overrule the model — that is the daily work of making AI output usable and trustworthy. It is distinct from building the models themselves, and it is the work I'm strongest at.
